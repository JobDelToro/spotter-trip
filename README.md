# Spotter Trip Planner

A full-stack application that generates HOS-compliant trip plans for property-carrying truck drivers. Given origin, pickup, and drop-off locations, the app calculates required rest breaks, fuel stops, and renders completed ELD daily log sheets.

**Live demo:** [spotter-trip.vercel.app](https://spotter-trip.vercel.app)

> **Note on cold starts:** The backend runs on Render's free tier and spins down after 15 minutes of inactivity. The **first request after inactivity may take up to 60 seconds** to respond while the service restarts. Subsequent requests are fast. This is a free-tier infrastructure constraint, not a code issue.

## Features

- **HOS compliance engine** — enforces FMCSA 11-hour driving / 14-hour duty window / 30-minute break / 10-hour reset rules for property-carrying vehicles on the 70-hour/8-day cycle
- **Fuel stop scheduling** — inserts mandatory fuel stops every 1,000 miles (0.5 hr duration)
- **Multi-day ELD log sheets** — SVG-rendered daily log sheets with accurate duty status blocks, automatically generated for the full trip duration
- **Interactive route map** — visualizes the full route polyline with annotated stop markers (pickup, drop-off, fuel, rest)
- **Location autocomplete** — real-time US city/address search

---

## HOS Assumptions

Per the assessment specification and FMCSA regulations:

- **Driver type:** Property-carrying
- **Cycle:** 70 hours / 8 days
- **Driving limit:** 11 hours per shift
- **Duty window:** 14 consecutive hours
- **Mandatory break:** 30 minutes after 8 hours of driving
- **Reset:** 10 consecutive hours off-duty
- **Fueling:** Every 1,000 miles (30 minutes)
- **Pickup / drop-off:** 1 hour each
- **Pre-trip inspection:** 15 minutes
- **Adverse conditions:** Not modeled

## API Endpoints

### `GET /api/locations/search/?q={query}`
Returns up to 5 US location results for autocomplete.

**Response:**
```json
[
  { "display_name": "Dallas, Texas, USA", "lat": 32.776, "lon": -96.796 }
]
```

### `POST /api/trip/plan/`
Calculates the full HOS-compliant trip plan.

**Request body:**
```json
{
  "current_location": "32.7762,−96.7968",
  "pickup_location": "29.7604,−95.3698",
  "dropoff_location": "34.0522,−118.2437",
  "current_cycle_hours": 10
}
```

**Response:**
```json
{
  "route": {
    "total_distance_miles": 1547.2,
    "total_duration_hours": 22.4,
    "geometry": [[32.776, -96.796], ...],
    "legs": [...]
  },
  "stops": [...],
  "eld_logs": [...],
  "summary": { ... }
}
```

## Local Development

### Backend

```bash
cd backend
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
python manage.py runserver
```

The API will be available at `http://localhost:8000`.

### Frontend

```bash
cd frontend
npm install
cp .env.example .env.local   # set VITE_API_URL=http://localhost:8000/api
npm run dev
```

The app will be available at `http://localhost:5173`.

## Deployment

### Backend — Render

The `render.yaml` at the root of the repo configures the backend as a Render Web Service. On first deploy:

1. Connect the GitHub repo in the Render dashboard via **New → Blueprint**
2. Render detects `render.yaml` automatically
3. Set any environment variable overrides in the Render dashboard if needed (e.g. `CORS_ALLOWED_ORIGINS` with your Vercel URL)

**Environment variables (set in Render dashboard):**

| Variable | Description |
|---|---|
| `DJANGO_SECRET_KEY` | Auto-generated by Render |
| `DJANGO_DEBUG` | Set to `False` |
| `DJANGO_ALLOWED_HOSTS` | Your Render service hostname |
| `CORS_ALLOWED_ORIGINS` | Your Vercel frontend URL |

### Frontend — Vercel

1. Import the repo in the Vercel dashboard
2. Set **Root Directory** to `frontend`
3. Add environment variable: `VITE_API_URL=https://<your-render-service>.onrender.com/api`
4. Deploy

The `vercel.json` handles SPA client-side routing rewrites automatically.

## Design Decisions

- **Stateless backend** — no database required; all computation is performed per-request, making the service trivially scalable and deployable
- **Photon over Nominatim** — Nominatim enforces a strict 1 req/sec policy and blocks shared-IP server traffic; Photon uses the same OSM dataset without rate limiting
- **OSRM public API** — provides accurate driving routes without requiring an API key; suitable for a demo/assessment workload
- **SVG ELD logs** — rendered server-side-computed, drawn client-side as SVG for pixel-perfect fidelity without external PDF libraries
